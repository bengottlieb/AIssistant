# Repository Guidelines

## Project Structure & Module Organization
- App source lives under `AIssistant/AIssistant/` with SwiftUI views in `Views/` and view models in `ViewModels/`.
- Core models and services are under `AIssistant/Internal/` (e.g., `Internal/Models/`, `Internal/Services/`).
- Assets are in `AIssistant/AIssistant/Assets.xcassets`.
- Build products are under `AIssistant/Products/` (generated by Xcode).
- Planning docs live in `AIssistant/Misc/`.

## Build, Test, and Development Commands
- Xcode: Product > Build (or Cmd+B) to compile the app.
- CLI (optional): list schemes with `xcodebuild -list`, then build with `xcodebuild -scheme AIssistant -configuration Debug`.
- Run in Xcode: select the AIssistant scheme and press Run (Cmd+R).

## Coding Style & Naming Conventions
- Swift/SwiftUI with 4-space indentation.
- Types use PascalCase (`CloudSyncService`), properties and methods use camelCase (`cloudStatusCache`).
- SwiftUI state uses `@State private var` and constants use `let`.
- Prefer async/await over Combine; avoid force unwrapping.
- Keep view code in `Views/` and business logic in `Internal/` services.

## Testing Guidelines
- No test targets are currently present. If adding tests, use Apple’s Testing framework for unit tests and XCUIAutomation for UI tests.
- Suggested naming: `FeatureTests.swift` in a `Tests` target, with test methods like `testLoadsContentList()`.

## Commit & Pull Request Guidelines
- Commit messages in this repo are short, lowercase, and descriptive (e.g., “added history”, “fixed item equality and hashing”). Keep them concise and action-oriented.
- PRs should include: a clear description of changes, linked issues (if any), and screenshots for UI changes.
- Keep PRs focused to a single feature or fix.

## Agent-Specific Notes
- Prefer Xcode tools for reading and editing project files.
- Limit changes to the requested task and avoid touching unrelated files.
